## 第四步:八条"踩过的坑"(必读)

这些是实战中踩出来的,再犯一次成本很高:

1. **Excel 行号敏感**:传导链路 Sheet 的 E 列公式**硬编码引用汇总 Sheet 的行号**。新增指标必须追加到末尾,不要在中间插入。
2. **IFS 函数不兼容**:所有打分公式用**嵌套 IF**,不用 IFS(LibreOffice 不认识)。
3. **操作 Excel 前先 unmerge**:合并单元格会阻止 openpyxl 写入。
4. **Python 读 Excel 必须先 recalc**:openpyxl 只能读缓存的公式值,如果 Excel 没被重新保存,公式值是旧的。脚本里有前置检查。
5. **计算型指标在 JSON 里没有 history**:如金油比、10Y-2Y 利差、社融累计等。画图时需要**实时计算**(从原始指标的 history 算衍生值)。
6. **日度数据别用单日做趋势判断**:南向资金、汇率这类,用累计值或"本月以来"变动。
7. **价格指标必须溯因**:PPI/PMI 上涨时,检查布伦特是否异动,若油价偏高必须标注"或部分受能源推动"。
8. **数值比较必须显式写数值**:写"A 高于 B"、"A 倒挂 B"时必须把两边数值写出来。项目历史真出过"1.475% 低于 1.18%"这种事故。

## 第五步:工作流程

### 用户让你改任何东西之前

1. 读 CLAUDE.md(入口)
2. 读 AGENT_PROTOCOL.md(工作协议)
3. 根据任务类型读对应 skill
4. **必做**:先给方案,用户确认后再动手。**不要先斩后奏**。

### 分阶段交付(重要)

多产物任务(Excel + Python + JSON + HTML + 文档)**不要一口气打包**:
1. 先发核心源文件(通常 Excel),等用户确认
2. 下游产物再做
3. 如果用户检查源文件要改,下游不用返工

### 沟通风格

- 用户是**非开发者**,解释技术问题要直白
- **语气温和克制**:禁用"暴涨、崩塌、恶化、一定、必然"等绝对化词汇,用"或反映、可能指向"替代断言
- **先讲方案、等确认、再动手**
- **遇到模糊的地方立即问用户**,不要自行猜测
- 每次改完都要更新对应 skill 和 changelog
