## 项目是什么

这是**投研团队的宏观仪表盘项目**,目标是把中美宏观数据转化为半自动化的投研报告。

**核心使用者是一位非开发者**——他使用 Wind 终端拉数据,需要 Codex/Agent 帮他完成所有涉及代码、公式、文档的工作。和他沟通时**先讲方案、确认、再动手**,禁止先斩后奏。

**项目的完整工作流(6 个环节,5 个已自动化)**:

```
① Wind 终端刷新数据 (人工)
    ↓
② Excel 公式自动更新各节点得分 (118 个指标 × 29 节点) — 全自动
    ↓
③ Python 脚本导出 JSON 快照 (完整版 + AI 用精简版) — 全自动
    ↓
④ AI 基于 Prompt 生成分析报告 (7 条传导链路) — 全自动
    ↓
⑤ HTML 研报模板渲染成页面 — 半自动(当前由 AI 手工编排)
    ↓
⑥ 飞书推送给团队 (⬜ 未开发)
```

## 核心硬规则(必记)

以下规则贯穿所有工作,任何场景下都不能违反:

### R1 沟通规则

**R1.1** 先讲方案,确认后再动手。不要先斩后奏。

**R1.2** 用户是非开发者。解释技术问题要直白,不要堆术语。

**R1.3** 语气温和克制。禁止用"极度、暴涨、崩塌、恶化、一定、必然、说明了、证明了"这些绝对化词汇。用"或反映、可能指向"替代断言。

### R2 数据分析规则

**R2.1 数据边界**:不能超越 JSON 已有数据做归因。总量数据(如"新增人民币贷款")不能拆成"企业融资意愿不足"这种分项判断,除非 JSON 里有分项指标。

**R2.2 日度数据不做趋势判断**:特别是南向资金这类波动大的指标,单日数据不能代表趋势。用累计值或月度变化。

**R2.3 价格指标必须溯因**:PMI/PPI 等价格相关指标上涨时,要检查是否是油价推动。如果当前油价偏高,必须标注"或部分受能源价格推动"。

**R2.4 术语首次出现必须通俗解释**:如 "DR007(银行间 7 天回购利率)"、"M2-M1 剪刀差(衡量资金活化程度)"。

**R2.5 数值比较硬规则**(2026-04-19 新增):分析里出现"A 高于 B"、"A 低于 B"、"倒挂"、"大于/小于"等数值比较表述时,**必须在稿子里显式写出两边的数值**,或在草稿阶段做快速算术验证。项目曾出现过"1.475% 低于 1.18%"这种事故。

**R2.6 TIPS 术语中文化**(2026-04-19 新增):用"10 年期通胀保值国债收益率",不用英文 TIPS。首次出现时加解释:"通胀保值国债的收益率扣除通胀补偿后 = 实际利率"。

### R3 技术规则

**R3.1 Excel 行号敏感**:传导链路 Sheet 的 E 列公式大量引用汇总 Sheet 的行号。改行号时必须同步改所有引用。

**R3.2 公式兼容性**:Excel 公式用嵌套 IF,不用 IFS(LibreOffice 不兼容)。用 `LOOKUP(9.99E+307, ...)` 取最新值,不用 `MATCH`(值重复会错)。

**R3.3 操作 Excel 前先 unmerge**:合并单元格会阻止写入,改动前必须先解除合并。

**R3.4 Python 导出脚本的 NODE_INDICATORS 映射和 Excel 行号绑定**:改 Excel 行号时一定要同步更新导出脚本。

**R3.5 Python 读 Excel 前 recalc**:openpyxl 只能读缓存的公式值。Excel 必须打开保存一次或用 LibreOffice headless recalc。脚本里有前置检查。

### R4 文档维护规则

**R4.1** 任何修改项目(Excel/Python/Prompt)后,必须同步更新对应的 skill 文档和变更日志。

**R4.2** 变更日志文件严格按命名:`YYYY-MM-DD_主题.md`(主题用简短英文 snake_case)。同日多轮独立存档。放在 `changelog/` 目录。

**R4.3** 如果发现新的"踩坑"或硬规则,加到本文档或对应 skill。

### R5 交付规则(2026-04-17 新增)

**R5.1 分阶段交付**:多产物任务(如 Excel + Python + JSON + HTML + 文档)**不要一口气打包发用户**。先发核心源文件(通常 Excel),等用户确认,再做下游产物。避免用户检查源文件要改时下游全部返工。

## 提示给新接手的 Agent

1. **不要假设你记得这个项目**。就算对话历史里有之前的信息,也要以本仓库文档为准。
2. **改动前先看 skill 文档**,里面有前人踩过的坑。
3. **每次改完都要更新文档**。这是硬性要求,不是可选项。
4. **遇到模糊的地方立即问用户**,不要自行猜测。用户的决策比你的推断重要。
5. **保持简洁**。用户时间有限,不要输出冗长废话。
6. **分阶段交付**:多产物任务先发核心文件,等确认再做下游。
7. **用 Codex 时**:每次 commit 前检查是否更新了 `changelog/`。
