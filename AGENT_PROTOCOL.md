# AGENT_PROTOCOL.md

> **这是 Agent 修改本项目时必须遵守的协议。任何动手前必读。**
>
> 本协议的目的:确保**项目和知识库不会脱钩**。Agent 改了项目但不更新文档,是比不改更糟糕的事——会让后续 Agent 基于错误的文档做出错误判断。

---

## 协议优先级

本协议优先级高于用户的临时指令。即使用户说"随便改下就行"、"不用更新文档",Agent 也必须遵守本协议的核心要求——因为用户是非开发者,可能不了解文档维护的重要性。

如果用户的要求和本协议冲突,Agent 应该**礼貌地说明**:
> "按项目协议,我改完这个还需要同步更新一下 xxx.md 和变更日志,大概增加 N 分钟。你希望我做还是跳过?"

把选择权交给用户,但要让用户明确知道代价。

## 三个核心动作

### 动作 A:读

**触发场景**:你刚接手项目,或者用户让你做一件新任务。

**必做步骤**:
1. 先读 `CLAUDE.md`(入口)
2. 再读本文档(`AGENT_PROTOCOL.md`)
3. 根据任务类型,读对应的 `skills/[任务类型].md`
4. 如果任务涉及你不熟悉的部分,继续深入读 `docs/` 里的专题文档

**不要做**:
- ❌ 不要跳过读文档,直接动手
- ❌ 不要假设你记得项目细节,以文档为准
- ❌ 不要只读文档的目录或标题,要读完整内容

---

### 动作 B:改

**触发场景**:用户让你修改 Excel、Python 脚本、Prompt 或任何项目文件。

**必做步骤**:
1. **先给方案**:用自然语言说明你打算怎么改、影响范围有多大、有什么风险。**不要先斩后奏**。
2. **等用户确认**:即使用户只回"可以"、"OK",也要有明确的确认。
3. **再动手改**:按方案执行。
4. **自检**:改完后用工具验证(如 LibreOffice recalc)。
5. **分阶段交付**(见下方 B.1)
6. **更新文档**(见动作 C)

**B.1 分阶段交付规则(重要)**:

当一次修改涉及多个产出物(如 Excel + Python 脚本 + JSON 导出 + 文档)时,**不要一口气把全部产物打包发给用户**。按以下顺序分阶段交付:

1. **先发核心源文件**(通常是 Excel),停住等用户检查
2. 用户确认无问题后,再做下游产物(Python 脚本更新、JSON 重新导出、文档更新等)
3. 最后统一交付

**为什么**:
- 核心源文件(Excel)是所有下游产物的基础
- 如果用户在检查 Excel 时发现问题要改,下游产物全部需要返工
- 分阶段交付避免这种返工成本,也让用户可以专注检查一个文件

**例外情况**:只有一个产出物、或产出物之间无依赖关系时,可以一次性交付。

**不要做**:
- ❌ 不要直接改,不给方案
- ❌ 不要方案和实际执行差太远(比如说改 3 个文件,结果改了 10 个)
- ❌ 不要静默修改多个相关文件却不提醒用户
- ❌ 不要在用户未检查核心文件前就完成下游所有产物

---

### 动作 C:记

**触发场景**:你完成了任何修改(改 Excel/Python/Prompt/文档本身都算)。

**必做步骤**:

**C.1** 同步更新对应的 skill 文档
- 改了 Excel → 更新 `skills/excel_maintenance.md`
- 改了 Prompt → 更新 `skills/analysis_writing.md`
- 加了指标 → 更新 `skills/indicator_management.md`
- 发现新"踩坑" → 加到对应 skill 的"踩坑记录"章节

**C.2** 在 `changelog/` 目录新建本轮的变更日志文件

**C.2.1 changelog 文件命名规范(重要)**:

文件名严格按以下格式,放在 `changelog/` 目录:

```
YYYY-MM-DD_主题.md
```

- **日期**:使用当天日期(年-月-日)
- **主题**:简短英文 snake_case,描述本次变更的性质,例如:
  - `excel_fix` — Excel 公式/格式的修复
  - `python_rewrite` — Python 脚本的适配或重写
  - `weight_refactor` — 权重体系重构
  - `prompt_update` — Prompt 的修改
  - `html_presentation` — HTML 展示页面
  - `doc_sync` — 纯文档同步
  - `doc_restructure` — 文档体系重构
  - `protocol_update` — 本协议的修改

**同一天多轮工作时,每轮单独一个文件**,即使是在同一个对话里产生的多份变更。不要把多段 changelog 挤在同一个文件里——即使同一天,不同性质的改动也应独立归档,便于后续查阅和追溯。

**错误示例**:
- `changelog_entry.md`(无日期无主题)
- `changes.md`(过于笼统)
- `2026-04-17.md`(同一天多轮会覆盖)

**正确示例**:
- `2026-04-17_excel_fix.md`
- `2026-04-17_weight_refactor.md`(同日两轮独立归档)

**C.2.2 变更日志内容模板**:

严格按以下模板,**不要自由发挥**:

```markdown
# [YYYY-MM-DD] 版本号或变更标题

**变更人**:Claude / Agent ID / 用户姓名
**变更类型**:[新增功能 / 修改现有 / 修复 Bug / 重构 / 文档更新 / 协议更新]
**影响范围**:
- 文件 1 的具体改动
- 文件 2 的具体改动

## 变更内容

(用 1-3 句话描述这次改了什么,为什么改)

## 同步更新的文档

- 更新了 xxx.md 的 xxx 章节
- 新增了 yyy.md

## 风险或遗留问题(如有)

- ...

## 下一步建议(如有)

- ...
```

**C.3** 如果改动影响了硬规则或引入了新规则,在对话里**主动告知用户**:
> "这次改动引入了一条新规则:xxx。我已经加到 CLAUDE.md 的 R 规则区,后续所有分析都会遵守。"

## 常见任务的处理流程

### 任务 T1:用户说"我加了几个指标到 Excel"

1. 读 `skills/indicator_management.md` 和 `skills/excel_maintenance.md`
2. 打开 Excel,搞清楚用户实际加了什么、加在哪里
3. **给方案**:告诉用户这些指标要挂到哪些节点、是主项还是子项、权重怎么分配、有没有打分公式冲突
4. 等用户确认
5. 改 Excel(包括汇总 Sheet 的公式 + 传导链路 Sheet 的节点公式)
6. 用 LibreOffice recalc 验证所有公式无 `#NAME?` `#REF!` 错误
7. 更新 Python 导出脚本的 NODE_INDICATORS 映射
8. 更新 `skills/indicator_management.md` 的指标字典
9. 写变更日志

---

### 任务 T2:用户说"分析的语气有问题,你调整下"

1. 读 `skills/analysis_writing.md`(里面有完整 Prompt 和写作规则)
2. 识别具体问题(是用词?结构?数据引用方式?数值比较错误?)
3. **给方案**:提出具体的修改规则,可能是加一条新硬规则到 Prompt 里
4. 等用户确认
5. 改 Prompt
6. 用样本数据测试一次,看 AI 产出是否符合新规则
7. 更新 `skills/analysis_writing.md`
8. 如果引入了新硬规则,同步更新 `CLAUDE.md` 的"核心硬规则"区
9. 写变更日志

---

### 任务 T3:用户说"我从新闻里看到 xxx,你觉得要不要加到项目里"

1. **先不急着改项目**。这类任务容易跑偏。
2. 先回答用户:这个信息可以怎么用,是加指标、改 Prompt、还是只作为参考
3. 如果涉及专业判断(比如是否要加一个新指标),参考专业机构研报的做法
4. 给用户**几个方案选**,不要只给一个
5. 用户选定后,按方案 T1 或 T2 执行

## 必须问用户(不能自己决定)的情况

以下情况 Agent **必须停下来问用户**,不能自行决定:

1. **修改会影响多条传导链路** — 比如改一个打分阈值会同时影响 A 股和港股
2. **删除现有指标或节点** — 可能有你不知道的依赖
3. **修改权重导致节点总权重不等于 1** — 可能是有意的,但要确认
4. **发现文档之间矛盾** — 比如 skill 说一个阈值,Prompt 里是另一个
5. **Python 脚本修改涉及新的依赖库** — 用户不一定有权限装新库
6. **Prompt 里要加新的硬规则** — 规则太多会让 AI 混乱,数量要控制

**标准询问格式**:

> 我注意到 xxx 情况。有两个方向可选:
> 方案 A:...
> 方案 B:...
> 你倾向哪个?或者有别的想法?

不要问"你觉得怎么办"这种开放式问题——用户是非开发者,可能不知道该怎么办。

## 文档之间的关系图

```
CLAUDE.md(入口)
    ↓ 指向
AGENT_PROTOCOL.md(本文档,定规则)
    ↓ 指向
skills/[任务类型].md(具体操作手册)
    ↓ 引用
docs/[专题].md(详细背景知识)
    ↓ 引用
changelog/YYYY-MM-DD_*.md(历史变更)
roadmap/pending_issues.md(未完成事项)
```

**写文档时:**
- 硬规则放 `CLAUDE.md`
- 操作流程放对应 `skills/*.md`
- 详细解释/背景知识放 `docs/`
- 历史变更放 `changelog/`
- 待办事项放 `roadmap/`

不要把操作流程写到 CLAUDE.md 里,也不要把背景知识写到 skill 里。**层次不能混。**

## 知识库自身的变更

如果修改的是知识库本身(比如优化了 skill 结构、修了一个 typo),也要记变更日志。标记 `变更类型:文档更新`。

如果修改涉及到**协议本身**(即本文档),必须:
1. 告知用户
2. 用户确认后才能改
3. 变更日志里专门标记 `变更类型:协议更新(重要)`

## 疑难场景

**场景 Q1:用户同时给了多个矛盾的要求**

比如先说"加 3 个新指标",后面又说"不要太多指标"。
处理:暂停,把矛盾明确告诉用户,让用户决定。不要自己调和。

**场景 Q2:用户指令不清晰**

处理:问澄清问题,但**一次最多问 3 个问题**,不要连环问。

**场景 Q3:用户的想法和专业机构做法相反**

比如用户觉得某个指标不重要,但多家券商都在用。
处理:礼貌地说明你看到的专业做法,但**尊重用户的最终决定**。用户可能有你不知道的考虑。

**场景 Q4:改动涉及非常多文件(>5 个)**

处理:拆成多步。先做一部分,让用户验证,再做下一部分。避免大批量静默修改。

**场景 Q5:数据/事实有疑问**

处理:**搜索验证**。不要基于记忆或推测给用户答案。项目曾出现过"1.475% 低于 1.18%"这种低级错误和"TGA 92.4 万亿"这种单位错误,都是本该搜索/验证但没做。

## Git 工作流(用 Claude Code 时)

当通过 Claude Code 操作本仓库时,每次修改后按以下顺序:

1. **修改文件**(按上面的动作 A/B/C)
2. **新建或更新 changelog/YYYY-MM-DD_主题.md**
3. **git add + git commit**,commit message 格式:`<主题>: 简短描述`,如:
   - `excel_fix: 修复 R88 银行贷款公式`
   - `doc_sync: 同步 indicator 字典`
4. **git push**

commit 的主题要和 changelog 文件名的"主题"一致,便于后续关联。

---

## 最后的原则

**宁可慢,不要错。** 用户的时间可以花在确认上,不应该花在排查 Bug 上。

**文档和项目是一体的**。改一个就要改另一个。否则知识库就成了废墟。

**保持谦虚**。你(Agent)只是项目的执行者,决策权在用户。

**数值必须验证**。出现任何数值比较、单位换算、字段对应时,都必须显式检查或写出两边数值。
